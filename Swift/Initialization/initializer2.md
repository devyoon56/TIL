# 클래스 지정 생성자와 편의 생성자

클래스의 지정 생성자는 구조체의 지정 생성자와 동일합니다. 클래스의 모든 저장 속성을 인스턴스 생성 시 초기화하는 역할을 맡는 것입니다. 클래스의 여러 생성자 중에 편의 생성자라는 것이 존재합니다. 편의 생성자는 편리하게 인스턴스를 생성하기 위한 보조 개념의 생성자라고 생각하면 됩니다. 결국 편의 생성자는 내부에서 지정 생성자를 호출하는 방식으로 모든 저장 속성을 초기화합니다.

따라서 편의 생성자는 모든 저장 속성을 초기화할 필요없이 지정 생성자를 호출하여 모든 저장 속성을 초기화합니다.

## 편의 생성자

- 지정 생성자보다 적은 개수의 파라미터로 인스턴스를 생성하는 보조 개념의 생성자입니다.
- 편의 생성자는 **convenience** 키워드를 사용하여 선언합니다.
- 편의 생성자는 지정 생성자를 호출하여 지정 생성자가 모든 저장 속성을 초기화하도록 합니다.

## 클래스 편의 생성자와 지정 생성자 예시

```swift
class Color {
    let red, green, blue: Double

    // 편의 생성자
    convenience init() {
        self.init(red: 0.0, green: 0.0, blue: 0.0)          // 지정 생성자 호출
    }

    // 편의 생성자
    convenience init(white: Double) {
        self.init(red: white, green: white, blue: white)    // 지정 생성자 호출
    }

    // 지정 생성자
    init(red: Double, green: Double, blue: Double) {
        self.red = red
        self.green = green
        self.blue = blue
    }
}
```

구조체에서 생성자가 지정 생성자를 호출하여 초기화하듯이 클래스에서 편의 생성자는 지정 생성자를 호출하여 모든 저장 속성을 초기화합니다.

## 편의 생성자 사용 이유

편의 생성자를 사용하는 이유는 클래스에 상속이라는 개념이 존재하기 때문입니다.

- 클래스는 상속을 지원하므로 상속 관계에서 저장 속성의 개수가 많아질 수 있습니다.
- 여러 지정 생성자를 구현했을 때 상속 관계의 복잡함으로 개발 실수의 가능성이 올라갑니다.

가능한 지정 생성자의 중복 구현을 최대한 없애고 편의 생성자에서 모든 저장 속성을 초기화하는 지정 생성자를 호출하는 패턴을 사용하는 것이 권장됩니다. 결국 편의 생성자에서 지정 생성자를 호출하는 것이 개발의 복잡도와 실수를 줄일 수 있는 방법입니다. 이 방법으로 편의 생성자는 초기화 과정을 간편하게 해줍니다.

## 클래스 상속과 편의 생성자, 지정 생성자 예시

```swift
class Aclass {
    var x: Int
    var y: Int

    init(x: Int, y: Int) {
        self.x = x
        self.y = y
    }

    convenience init() {
        self.init(x: 0, y: 0)       // 지정 생성자 호출
    }
}

// Aclass 상속
class Bclass: Aclass {
    var z: Int

    init(x: Int, y: Int, z: Int) {
        self.z = z                  // 현재 클래스의 저장 속성 먼저 초기화
        super.init(x: x, y: y)      // 슈퍼 클래스의 지정 생성자를 호출하여 상속한 저장 속성 초기화
        self.x = 10
        self.y = 10
    }

    convenience init(z: Int) {
        self.init(x: 0, y: 0, z: z)     // 지정 생성자 호출
    }

    convenience init() {
        self.init(z: 0)                 // 다른 편의 생성자 호출
    }
}

let a1 = Aclass(x: 1, y: 1)          // 지정 생성자로 인스턴스 생성
a1.x        // 1
a1.y        // 1

let a2 = Aclass()                   // 편의 생성자로 인스턴스 생성
a2.x        // 0
a2.y        // 0

let b1 = Bclass(x: 1, y: 1, z: 1)       // 지정 생성자로 인스턴스 생성
b1.x        // 1
b1.y        // 1
b1.z        // 1

let b2 = Bclass(z: 2)                   // 편의 생성자로 인스턴스 생성
b2.x        // 0
b2.y        // 0
b2.z        // 2

let b3 = Bclass()                       // 편의 생성자로 인스턴스 생성
b3.x        // 0
b3.y        // 0
b3.z        // 0
```

위 코드를 보면 편의 생성자로 인스턴스를 생성하여도 내부적으로 지정 생성자를 호출하여 인스턴스를 생성합니다. 그리고 편의 생성자에서 다른 편의 생성자를 호출할 수도 있습니다. 다른 편의 생성자를 호출하더라도 결국 호출된 편의 생성자는 지정 생성자를 호출하여 인스턴스를 생성한다는 것을 알 수 있습니다.

또한 Bclass의 지정 생성자에서 주의할 점은 슈퍼 클래스의 지정 생성자를 호출하기 전에 반드시 Bclass의 모든 저장 속성을 먼저 초기화해야 한다는 점입니다. 그리고 Bclass의 지정 생성자에서 슈퍼 클래스 Aclass의 지정 생성자를 호출하여 상속한 모든 저장 속성을 초기화한다는 것도 알 수 있습니다.

위 코드는 아래에서 살펴볼 규칙을 따른 것입니다.

## 클래스 상속과 지정 생성자, 편의 생성자의 호출 규칙

클래스 상속 관계에서 지정 생성자와 편의 생성자를 호출하는 규칙이 존재합니다. 이 규칙을 생성자 위임 규칙(Initializer Delegation)이라고 합니다. 이 규칙에 주의해서 지정 생성자와 편의 생성자를 호출해야 합니다.

1. 델리게이트 업(Delegate Up)

- 서브 클래스의 지정 생성자는 슈퍼 클래스의 지정 생성자를 반드시 호출해야 합니다.
- 서브 클래스의 저장 속성을 먼저 초기화한 후 슈퍼 클래스의 저장 속성도 초기화합니다.(상속한 저장 속성)

2. 델리게이트 어크로스(Delegate Across)

- 편의 생성자는 동일한 클래스에 있는 다른 편의 생성자를 호출하거나 지정 생성자를 호출해야 합니다.
- 궁극적으로 반드시 지정 생성자를 호출해야 합니다.

이 규칙을 지키지 않으면 인스턴스가 올바르게 메모리(힙 영역)에 생성되지 않습니다. 인스턴스 생성을 완료하려면 모든 저장 속성이 반드시 초기화되어야 합니다.

## 클래스의 상속과 초기화 과정의 이해

생성자 위임 규칙을 사용하여 클래스의 상속과 초기화 과정을 살펴보겠습니다.

기본 개념은 다음과 같습니다. 인스턴스의 생성을 완료하려면 인스턴스의 모든 저장 속성이 초기값을 가져야합니다. 그리고 현재 클래스에서 선언된 저장 속성은 반드시 해당 클래스에서 초기값을 가져야 합니다. 이것은 서브 클래스의 저장 속성은 서브 클래스에서, 슈퍼 클래스의 저장 속성은 슈퍼 클래스에서 초기화되어야 한다는 의미입니다. 다르게 표현하자면 각 클래스의 저장 속성은 각 클래스의 지정 생성자를 통해 초기화되어야 한다는 의미입니다.

1단계는 필수 사항입니다.

- 먼저 현재 클래스에서 선언한 모든 저장 속성을 초기화합니다.
- 슈퍼 클래스의 지정 생성자를 호출하여(델리게이트 업) 슈퍼 클래스의 모든 저장 속성을 초기화합니다.
- 최종적으로 기본 클래스(상속 단계의 최상위 클래스)의 모든 저장 속성에 값이 있는지 확인되면
- 상속 관계의 모든 저장 속성이 완전히 초기화된 것으로 간주되어 인스턴스의 생성이 완료됩니다.

2단계는 선택 사항입니다.

- 인스턴스가 생성된 후
- 상속 관계의 가장 위에서 아래로 내려가면서 인스턴스를 생성하기 위해 호출된 각 단계의 생성자가 인스턴스를 수정할 수 있습니다.

인스턴스를 수정한다는 의미는 인스턴스에 저장된 속성의 값을 바꾸거나 메서드를 호출할 수 있다는 의미입니다. 인스턴스가 생성된 이후이므로 self 속성에 접근할 수 있기 때문에 속성을 수정할 수 있고 인스턴스의 메서드를 호출할 수 있습니다.
